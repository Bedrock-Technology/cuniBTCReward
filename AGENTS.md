# AGENTS.md — cuniBTCReward

## Project Summary

Go service for managing cuniBTC reward airdrops: scans EVM chains for vault events, calculates position-weighted APY/averages, builds Merkle trees for trustless claims, and serves a REST API (go-zero).

## Build & Code Generation

```bash
# Build binary
go build -o cuniBTCReward .

# Regenerate go-zero handlers/types from API definition
cd api && goctl api go -api cuniBTCReward.api -dir .

# Regenerate Swagger docs
cd api && goctl api swagger -api cuniBTCReward.api -dir docs

# Regenerate contract Go bindings (per contract)
cd service/contract && abigen --abi ./<contract>/<contract>.json --pkg <contract> --out ./<contract>/<contract>.go
```

Run tests: `go test ./...`

## Architecture: Handler → Logic → ServiceContext

All API endpoints follow the go-zero pattern:
```
Handler (parse HTTP) → Logic (business rules) → ServiceContext (DB, cron)
```

- **Handlers** in `api/internal/handler/` — wire into routes via `routes.go`
- **Logic** in `api/internal/logic/` — contains actual business logic, receives `*svc.ServiceContext`
- **ServiceContext** in `api/internal/svc/servicecontext.go` — GORM DB handle, cron schedulers, shared state

Response format (see `api/response/response.go`):
```json
{"code": 0, "msg": "ok", "data": {...}}
```

## Database (GORM + MySQL)

- All entities in `model/model.go` — use `gorm.Model` (ID, CreatedAt, UpdatedAt, DeletedAt)
- **Always use `decimal.Decimal`** for monetary amounts — never float64
- Unique indexes use the `index:...,unique` GORM tag convention
- Custom GORM logger in `pkg/gormz/gorm.go` — integrates with go-zero's `logx`
- Database schema matches models exactly (see `model/sql.sql`)

## CLI (Cobra)

Three run modes via `cmd/`:
```bash
./cuniBTCReward api --config config.yaml          # REST server
./cuniBTCReward airdrop shares ...                 # Calculate airdrop shares
./cuniBTCReward airdrop create ...                 # Build Merkle tree & persist epoch
./cuniBTCReward evmscan --config config.yaml       # Background chain scanner
```

Config aggregation in `cmd/root.go` (`ServiceConfig` struct).

## Smart Contracts

Contract bindings auto-generated by `abigen`. Four contracts:
- **CuniBTCVault** — deposit/withdraw events
- **DelayRedeemRouter** — delayed redemption events
- **Airdrop** — Merkle root storage & claims
- **Factory** — strategy creation events

Regenerate with commands in `service/contract/README.md`.

## Key Conventions

- **Chain IDs**: 560048 = Hoodi testnet (used in `config-hoodi.yaml`)
- **Money math**: Always `decimal.Decimal`, never `float64`
- **RPC rate limiting**: Configured per-chain via `clientrpc/goeth/client.go` (`ThrottledTransport`)
- **Slack alerts**: Configurable via `LogSlack` / `NotifySlack` in yaml; logged via `pkg/slack/slack.go`
- **Cron specs**: Standard cron expressions (e.g., `@every 12s`, `@every 30m`)
- **Merkle trees**: Use `github.com/FantasyJony/openzeppelin-merkle-tree-go` (standard OpenZeppelin format)
